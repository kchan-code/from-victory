## PR-04 — the decision pending

`feature/athlete-account-age-floor` is the next critical-path PR. **Three real auth-model options. None has been chosen yet.** All are workable; pick one before coding.

The constraint: **athletes do not have email addresses** (CLAUDE.md non-negotiable). Supabase Auth's default sign-in requires an email + password or magic link. So we need a workaround.

| Option | How athletes authenticate | Trade-offs |
|---|---|---|
| **A. Synthetic email** | Server generates a `athlete-{uuid}@athletes.fromvictory.app` address; password set by parent. Athlete signs in with their first name + password (UI maps name → synthetic email server-side). | Simplest mental model. Supabase Auth used as-is. Email never shown to athlete. Requires picking a domain (can be made-up; nothing has to receive mail). |
| **B. Parent-set passcode + custom session** | Parent sets a 4-6 digit passcode for each athlete. Athlete signs in by picking their name + entering passcode. Server validates passcode, mints a JWT using service role. | No synthetic email. More custom code (custom passcode-check server action, JWT minting). Aligns with how kids' devices/games handle auth. |
| **C. Personal magic-link / QR per athlete** | Parent generates a one-time link or QR code per athlete; athlete clicks/scans on their device, gets a long-lived session token. | Simplest UX once set up. Token rotation strategy needed. Best if athletes mostly use a single device (true for MVP PWA-on-phone case). |

Recommendation: **A (synthetic email)** for MVP simplicity. It plays nicely with Supabase Auth without custom JWT minting, the email is server-only (never surfaced), and it gives every athlete a stable `auth.users` row that everything else hangs off. KC should confirm before code starts.

PR-04's other concerns (independent of auth model):
- Parent signup flow (real email + password — standard Supabase Auth).
- Birthdate captured at athlete creation; **reject < 13** at the server-action level (a Zod refine), since the DB only enforces presence not minimum.
- `parent_athlete_links` row created via service-role server action.
- Supabase client wrappers in `apps/web/lib/supabase/server.ts` + `client.ts`. New deps: `@supabase/ssr` + `@supabase/supabase-js` (needs product-strategist approval per CLAUDE.md but they're industry standard).
- `next/middleware.ts` for refreshing the session cookie on every request.

Probably split as **PR-04a** (wrappers + deps + middleware, no UI) and **PR-04b** (signup flows + 13+ floor) for cleaner review.
